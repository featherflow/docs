# Bitscale Workflow Model

Here's the insight that separates power users from everyone else: **a Grid is not a spreadsheet. It's a pipeline.**

A spreadsheet holds data. A Bitscale Grid *processes* data. Every time a new row arrives, columns fire. Every time you update a run condition, the behaviour of the entire workflow changes. Understanding this model unlocks everything that comes after Foundation.

## Sources: How Data Enters a Grid

There are four ways data gets into a Grid:

| Source Type | How It Works | Best For |
|-------------|-------------|---------|
| **CSV Upload** | Manual import from a file | One-off lists, test batches |
| **Live Sync** | Connected to a CRM, tool, or API that pushes rows automatically | Ongoing workflows |
| **Webhook** | External trigger sends a new row to your Grid | Product signals, form submissions, intent events |
| **Manual Entry** | Type rows directly | Quick tests, very small batches |

The moment a row enters a Grid, all active column enrichments fire on it automatically. This is what makes Live Sync sources powerful — the pipeline runs continuously without anyone pressing a button.

## Columns: Where the Work Happens

Columns are the engine of every Grid. There are three types:

| Type | What It Does | Example |
|------|-------------|---------|
| **Static (Text)** | Holds data entered manually or from import | Company Name, Domain |
| **Enrichment** | Calls an external API and returns a value | Headcount, Funding Data, Contact Finder |
| **AI** | Sends data to an LLM and returns generated content | Email opener, ICP score, research summary |

Each column has a **run condition** that controls when it fires. This is the most underused feature in Bitscale — and the one that separates efficient workflows from expensive ones.

### Run Conditions

**Always:** Fires every time a row is added or updated. Good for fast, cheap enrichments you always need.

**On condition:** Fires only when another column meets a specific threshold. Example: only run Contact Finder if `SDR Hiring Signal > 0`. This saves credits and keeps the workflow clean.

**Manual only:** Never fires automatically. You trigger it row-by-row or in batches. Good for expensive or sensitive enrichments you want to review before running.

**Example: A cost-efficient workflow using run conditions**

```
Column: Headcount         → Always         (cheap, always useful)
Column: SDR Hiring Signal → Always         (cheap, key filter)
Column: Contact Finder    → On condition: SDR Hiring > 0
Column: AI Email Opener   → On condition: Contact is not empty
Column: ICP Score         → On condition: All above are populated
```

Result: You only spend credits on expensive columns for rows that have already passed your ICP filters. On a 200-row grid, this can save 60–70% of your monthly credit usage.

## Exports: Where Enriched Data Goes

When your columns are populated, data leaves via an Export.

| Export Type | How It Works |
|-------------|-------------|
| CSV download | Manual export on demand |
| CRM push | Sync to HubSpot, Salesforce, Attio automatically |
| Sequencer push | Send contacts directly to Outreach, Apollo, Smartlead, Instantly |
| Webhook | POST to any endpoint — Slack, custom APIs, Zapier |
| Scheduled auto-export | Triggers on a time interval without manual action |

You can add conditions to exports: `Only export rows where ICP Score > 7 AND Email is verified`. This means only your best-matched, reachable contacts ever leave Bitscale.

## Scheduling: Making It Autonomous

Grids can run on a schedule. This turns a one-off workflow into a continuous GTM motion.

**Example: Automated weekly SDR-hiring intent scan**

1. **Source:** LinkedIn Sales Navigator saved search (your ICP filters)
2. **Grid:** Auto-imports new results every Monday at 6am
3. **Columns:** Headcount → Hiring Signal → Contact Finder → AI Opener → ICP Score
4. **Export:** Auto-push to Outreach sequence if `ICP Score > 7 AND Contact found`

Every Monday, your SDRs start the week with 15–20 fresh, enriched, scored prospects already sitting in their sequence tool. Nobody pressed a button.

**That's what GTM Automation looks like when it's fully operational.**

## The Full Mental Model

```
[Source: LinkedIn / CSV / CRM / Webhook]
                ↓
[Grid: Your processing environment]
                ↓
[Column 1: Company Name]     Static
[Column 2: Domain]           Static
[Column 3: Headcount]        Enrichment → Always
[Column 4: Hiring Signal]    Enrichment → Always
[Column 5: Contact Finder]   Enrichment → On condition: Hiring > 0
[Column 6: AI Opener]        AI         → On condition: Contact found
[Column 7: ICP Score]        AI         → On condition: All above populated
                ↓
[Export → Outreach / HubSpot / CSV]
[Condition: ICP Score > 7 AND Email verified]
```

Every workflow you'll ever build follows this pattern. The complexity lives in what's inside the columns and which conditions you set. The architecture stays the same.
